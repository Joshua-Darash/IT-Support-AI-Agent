Core Architecture
Agent Reasoning Loop
1
💭
THOUGHT
The agent receives the incident description and forms an internal reasoning plan. It identifies what type of incident this is, which tools are relevant, and what information it needs to gather before recommending any action.
Incident classification Tool planning
2
⚙️
ACTION
The agent selects and calls the most appropriate tool — always starting with Runbook Search to ground its response in known procedures. It then calls Log Search, Server Metrics, or Status Check based on what the runbook suggests or what the incident requires.
Tool selection Parameterised calls
3
👁️
OBSERVATION
Tool outputs are returned as structured JSON and fed back into the agent context. The agent reads the evidence — log errors, metric values, service states — and incorporates it into its understanding of the incident. Multiple rounds of tool use may occur before enough evidence is gathered.
Evidence gathering Grounding
4
🔄
REFLECTION
Using all gathered observations, the agent produces a structured diagnostic output: incident type, root cause (evidence-backed), recommended remediation steps sourced from the runbook, and severity. It then calls Create Ticket to formally log the incident, which flows to Trello.
Root cause analysis Runbook citation Ticket creation
Tool Orchestration
Integrated Tools
📖
Runbook Search
RAG · KB-001 through KB-007
Keyword-based semantic search across 7 embedded IT runbooks covering disk, VPN, app crashes, web performance, email sync, and more. Called first on every incident to ground the agent in documented procedures and avoid hallucination.
query Natural language description of the incident
📋
Log Search
Simulated log database
Returns timestamped log entries for 5 servers (nginx, vpn, app, mail, disk). Supports filtering by service, log level (ERROR/WARN/INFO), and time range. Surfaces OOMKilled, crash loops, connection failures, and sync errors.
servicenginx · vpn · app · mail · disk
levelERROR · WARN · INFO · all
since1h · 6h · 24h
📊
Server Metrics
5 simulated servers
Returns CPU %, memory %, disk %, uptime, load average, and overall status (healthy / degraded / critical / down) for web-01, db-01, vpn-01, mail-01, app-01. Enables quantitative evidence in the agent's diagnosis.
serverweb-01 · db-01 · vpn-01 · mail-01 · app-01
🔍
Status Check
Service health states
Returns live service state including PID, port, uptime, restart count (24h), last restart time, and health check result. Surfaces restarting / degraded / down states and maps them to the correct server and service pair.
servicenginx · vpn · app · mail · database
serveroptional target server

INC-XXXX · Trello integration
Creates a structured incident ticket with a unique ID (INC-XXXX), severity-based SLA target, full diagnostic description, and assigned team. The ticket then flows downstream through Extract Ticket Fields → Format Ticket Data → Create Trello Card, creating a real card on a Trello board for tracking.

Retrieval & Memory
RAG Pipeline & Context
RAG — Runbook Grounding
Approach
Keyword-based semantic search across 7 embedded runbooks. No external vector store — fully self-contained inside the n8n Code node.
Runbooks
KB-001 Disk · KB-002 VPN · KB-003 App Crash Loop · KB-004 Web Perf · KB-005 Email · KB-006 DB · KB-007 Security
Grounding rule
Agent always calls Runbook Search before any remediation recommendation. Citations are included in the output (e.g. "per KB-003, step 4...").
Hallucination prevention
System prompt explicitly instructs: "Do not recommend actions not covered in the retrieved runbook." If no runbook matches, the agent escalates instead of guessing.
Memory — Conversation Buffer
Type
Window Buffer Memory — retains the last 5 conversation turns (configurable), attached to the LangChain agent node.
Session scoping
Each session uses a unique sessionId generated in the React UI (UUID). Memory is scoped to that session, so parallel users stay isolated.
What is stored
User messages and agent responses. Tool call inputs/outputs are not persisted in memory between turns — only the final response.
Sensitive data policy
Credentials, tokens, and PII are never stored. Memory is ephemeral (in-process). No disk or database persistence.
Safety Architecture
Safety Controls
🛡️ Prompt Injection Detection
  • Regex-based pattern matching on all incoming messages before they reach the agent
  • Detects: "ignore previous instructions", "act as unrestricted", "reveal system prompt", "delete all tickets", "bypass safety"
  • Blocked inputs return a canned refusal message and never reach the LLM
  • Block reason is logged for observability
// Sample injection patterns
const PATTERNS = [
  /ignore (previous|all) instructions/i,
  /reveal (your )?(system prompt)/i,
  /delete all (tickets|data)/i,
  /bypass (safety|restrictions)/i
];
🔒 Role-Based Access
  • Admin — Full access including config and webhook URL management
  • IT Agent — Chat, history, create tickets
  • Viewer — Read-only chat access, cannot modify config
  • Role is enforced in the React UI; all users must authenticate before reaching the webhook
🚫 Tool Misuse Prevention
  • Create Ticket is only called by the agent after a complete diagnosis — not speculatively
  • Severity is agent-inferred from evidence, not user-supplied directly
  • Assigned team is restricted to a whitelist of 4 known teams
  • No destructive tool actions exist (read-only metrics, logs, status)
🔐 Sensitive Data Protection
  • No credentials are stored in memory or tickets
  • n8n credentials (OpenAI key, Trello token) stored in n8n's encrypted credential vault — never in code
  • Demo credentials in the UI are clearly marked and would be replaced by SSO in production
  • No PII is logged in the observability trace
Observability
Trace & Observability

The Observability Trace Logger node runs in parallel after the agent completes. It captures tool calls, inputs, outputs, reasoning steps, and decision transitions — making the agent's reasoning fully visible and non-black-box.

1
THOUGHT
Incident classified as "App service restarting repeatedly". Relevant tools: runbook_search, log_search (app), server_metrics (app-01), status_check (app).
2
ACTION → OBSERVATION
runbook_search({ query: "service restarting crash loop memory" })
→ KB-003 matched: App Crash Loop Recovery. Trigger: restarts > 5 in 1h. Diagnosis: check memory OOM, container limits, heap config.
3
ACTION → OBSERVATION
server_metrics({ server: "app-01" })
→ CPU: 95%, Memory: 99%, Disk: 62%, Status: CRITICAL, Uptime: 12 minutes
4
ACTION → OBSERVATION
log_search({ service: "app", level: "ERROR", since: "1h" })
→ [10:45:23] OOMKilled: container exceeded 2Gi · [10:45:20] OutOfMemoryError: Java heap space · [10:45:10] Service restarting (attempt 7 of 10)

→ INC-4721 created · SLA: 1 hour · Trello card published
Reflection
Design Trade-offs
✅ What Worked Well
✓n8n's native LangChain agent node provided the ReAct reasoning loop out of the box, letting design effort focus on tool quality and safety rather than loop plumbing.
✓Embedding runbooks directly in the Code node (rather than an external vector store) made the RAG layer portable, fast, and zero-dependency — a pragmatic choice for a demo environment.
✓The Safety Layer running as a pre-agent Code node means injection attempts never consume LLM tokens and can't influence the system prompt.
✓Trello integration demonstrates a real side-effect action, distinguishing the system from a pure chatbot.
⚠️ Known Trade-offs & Next Steps
→Simulated tool data: All metrics, logs, and service states are dummy data. In production these would call real APIs (Datadog, Splunk, PagerDuty). The reasoning logic remains identical.
→RAG scaling: Keyword search doesn't handle synonyms well. Production upgrade: embed runbooks into a vector store (e.g. Pinecone) for semantic similarity matching.
→Memory scope: Window buffer resets when n8n restarts. Production: persist session memory to Redis or PostgreSQL for cross-session continuity.
→Auth: Demo credentials are hardcoded in the UI. Production: replace with OAuth / SSO and server-side session management.
